# 텍스트

### 두 가지 이상 서체를 사용하지 마라

- 서체와 폰트의 차이점
    - 서체: 글자 모양 자체
    - 폰트: 서체를 컴퓨터에 구현한 파일
- 제목용, 본문용으로 최대 2가지 폰트를 사용
    - 서체를 지나치게 다양하게 사용하면 사용자의 인지에 방해가 된다.

### 시스템 폰트를 사용해야하는 이유

1. 웹사이트 로딩 속도 증가
2. 글자가 선명하게 출력
3. 가독성 향상
4. 모든 사용자에게 동일한 경험 제공

만약 시스템 폰트를 사용하지 않는다면...

- 폰트를 다운로드해야 하기 때문에 추가적 시간 발생
- 폰트가 렌더링되는동안 깨지는 문제
- 디바이스별로 다른 이슈 대응 방식
- 반응형 디자인 시 차이 발생

### 본문에 합리적인 기본 크기를 사용하라

기본 크기를 찾기 전, 고려할 사항

1. 전세계적 저시력 인구는 약 1억 2천만명
2. 노인층의 증가에 따른 시각저하 인구 증가
3. 편하게 읽기 좋은 컴퓨터 화면은 20 - 23인치
4. 인쇄물보다 화면에 보여지는 텍스트가 사용자와의 거리감이 더 멀게 느껴진다.
- 플랫폼별 권장 폰트 사이즈
    - 안드로이드 권장 최소 폰트 사이즈: 16sp
    - iOS 가이드 권장 최소 폰트 사이즈: 17pt
    - WCAG 2.0 표준 최소 폰트 사이즈: 18pt

### 행간은 x1.5?

본문 폰트에서 최적의 가독성을 위해 1.5배를 권장하는 것일 뿐, 필수는 아니다.

- e.g) google material guide에서도 폰트 사이즈, 행간 사이즈를 각각 14ps - 20dp, 20ps - 28dp를 권장하고 있다.

서비스에 알맞은 비율로 설정하는 것이 중요!

# 플랫 디자인

의 단점으로 지적되어 왔던 것들 (출처: [https://ppss.kr/archives/80638](https://ppss.kr/archives/80638){: target="_blank"})

- 어떤 부분이 클릭 가능한 것인지 구분하기 모호해짐
- 단순한 스타일 때문에, 서로 다른 사람이 만든 디자인이라도 비슷해보일 수 있음
- 과도한 유행 e.g) 아무 이유 없이 사용되는 긴 그림자

이를 보완해서 만든 것으로 '플랫 2.0' (혹은 almost-flat 이라고도 불린다고 함)이 있고, 구글의 머터리얼 디자인과도 유사점이 많은 디자인이다.

---

아래 내용은 플랫 디자인에서 지키면 좋은 항목들이다.

### 줄임표를 사용해 다음 단계 알리기

줄임표는 추가 단계가 필요하다는 것을 사용자에게 암시한다.

- 단, 이 때 그냥 마침표 3개를 찍지 말고 특수문자를 사용하자. 서체에 구애받지 않고 같은 결과를 보여준다.

### 버튼을 버튼처럼 보이게

여기선 지킬 것들이 꽤 있다.

- 텍스트보다 컨테이너가 작으면 안 된다. (당연히)
- 아이콘과 텍스트를 세로로 정렬하지 말 것 (이에 대한 특별한 이유는 아직 모르겠다.)
- 2가지 이상 아이콘을 버튼 하나에 넣지 말 것
- 텍스트 버튼과 컨테인드 버튼 (배경이 칠해져있는 버튼)을 세로로 나란히 두지 말자.
- 너무 긴 텍스트를 사용하지 말 것.

### 적절한 크기의 버튼을, 기능에 따라 그룹핑

- 피츠의 법칙: 타겟이 재빨리 이동하는데 걸리는 시간은 타겟까지의 거리와 **타겟의 크기**에 따라 달라진다.
- 마이크로소프트 연구 결과, 디바이스에 상관없이 최소 크기 7 x 7 mm를 사용하는 것이 좋다고 한다.
    - 안드로이드 가이드 권장 최소 사이즈: 36px
    - iOS 가이드 권장 최소 사이즈: 44pdi
- 같은 그룹 내의 버튼이라도 너무 가깝게 배치하지 말자.
    - 구글에서는 터치 대상에게 가로, 세로 최소 8px 이상의 여백을 주라고 한다.

### 텍스트만이 아니라 버튼 전체가 클릭 가능하게

- 개발 할 때 흔히 하는 실수..!
    - 버튼의 어느 부분을 눌러도 작동해야한다.
- 버튼의 **시각적 피드백**을 사용자에게 제공할 것
    - e.g) 마우스를 올렸을 때 굵기나 색상 변화
